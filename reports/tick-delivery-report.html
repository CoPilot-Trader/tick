What the Client Asked For

Per the Statement of Work v2, WhosTheBoss commissioned a multi-agent stock market price prediction system with the following high-level characteristics:

  • A Next.js dashboard that presents live prices, predictions, and trading signals to end users who plan to invest real money based on the tool.
  • A FastAPI backend coordinating a fleet of specialized AI agents covering data ingestion, feature engineering, price forecasting, trend classification, support/resistance detection, sentiment analysis, signal fusion, backtesting, and alerting.
  • A real-time pipeline — OHLCV candles, predictions, and sentiment — streamed to the frontend with low latency.
  • A Dockerized deployment posture compatible with GCP Cloud Run and on-prem servers.
  • A testing discipline covering unit, integration, and end-to-end flows with ≥ 70% coverage on the API layer at the M5 checkpoint.

User's core "why"

Because people will invest real money based on the predictions this tool surfaces, accuracy and visible trustworthiness are the guiding principles. Every design decision — the prediction backtracking overlay, the walk-forward validation harness, MAPE/directional accuracy stats in the chart toolbar — is in service of making the gap between predicted vs actual visible.

Additional Guarantees Delivered Beyond the SoW

  • Prediction backtracking store. Every prediction made by the platform is persisted per-symbol so that over time the user can visualize how accurate each prediction was — not in the SoW, added because it reinforces the user's core "why".
  • Parallel auto-prediction across all 31 tracked tickers every 5 minutes in the background — the platform is always ready with a fresh forecast when a user loads the dashboard.
  • Multi-timeframe fusion (1h / 1d / 1w) exposed through a single API call.
  • News markers on the chart — every article fetched shows a dot on the candlestick with a hover tooltip.
  • M5 sign-off document — in addition to the HTML reports, a markdown signoff is committed under docs/MILESTONE_SIGNOFF_M5_Frontend_Testing.md.

Cloud Run Deployment Recipe

The reports/ folder is a self-contained nginx site. To publish it at a shareable URL:

# 1. Authenticate + select your GCP project
gcloud auth login
gcloud config set project YOUR-PROJECT-ID

# 2. Build and deploy the reports site (from repo root)
cd reports && ./deploy.sh

# 3. The script prints the public URL when done
# e.g. https://tick-reports-xxxxxx.us-central1.run.app

The deploy script uses gcloud run deploy --source . so there is no intermediate registry push. Allow unauthenticated access is enabled by default so the client can share the URL with anyone.

Strategy 2 — PCR Shock Forecast (Predicted vs Actual)

Tory's Put-Call Ratio Shock detector: when the system spots unusual options activity, it predicts price direction at multiple horizons (15m, 30m, 1h, 2h, 1d) and the tracker then captures actual moves. This fits the existing "Pred vs Actual" backtracking panel exactly — same shape, different source. 59% accuracy on 942 resolved signals.

Backend reshapes PCR data into the same PredictionHistoryEntry format used by the M5 backtracking pipeline. Entries get a source: "pcr_shock" tag so they can be filtered or color-coded later, but they merge transparently into the existing chart.

Known Limitations

Tiingo 403 on free tier. The bundled Tiingo collector requires a paid API key for real-time data on the free tier it returns 403. The Data Agent gracefully falls back to yFinance, then to mock, so no user-facing breakage — but real-time intraday on new tickers benefits from a paid Tiingo key.
OpenAI key not bundled. llm_sentiment_agent works out-of-the-box with a deterministic mock client. Supply OPENAI_API_KEY in .env to enable real GPT-4 scoring — we recommend the 4o-mini model for ~10× lower cost.
Alert notification handlers. Alerts are stored in-memory and served via API. External fan-out (email / Slack / SMS) is a one-file extension — the AlertAgent accepts notification handlers as a plug-in point.
In-memory alert history. Alerts don't persist across backend restarts. Moving to TimescaleDB is a ~1-day task when you're ready.
No admin UI. Rule enable/disable, weight tuning, model retraining are API-driven. A small admin page (8–12 hours) would make this one-click.
Streaming loop uses a 5-second poll. This is adequate for equities but crypto / forex users would benefit from a push model (webhook → Redis pub/sub). Swap in without touching the frontend.
Mobile-web dashboard. The dashboard is responsive and works on mobile but isn't native. A React Native or Flutter wrapper is the right Phase-2 move.
